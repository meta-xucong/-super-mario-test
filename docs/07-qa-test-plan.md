# QA 与测试计划

状态：基线

## 测试层级

| 层级 | 目标 | 例子 |
| --- | --- | --- |
| 单元 | 独立规则正确 | 重力、扫掠碰撞、计分去重、输入边沿 |
| 数据 | 关卡清单合法 | ID 唯一、入口有出口、出生点安全、终点唯一 |
| 集成 | 系统协作正确 | 顶砖生成道具、踩敌反弹、进管转场、死亡重置 |
| 回放 | 确定性不回退 | 同输入、同版本产生同快照与事件序列 |
| 视觉 | 可读性和布局未漂移 | 关键帧截屏、HUD 遮挡、碰撞调试图 |
| 探索 | 边界与设备质量 | 焦点丢失、低帧、手柄断连、不同浏览器 |

## 必须自动化的用例

每次合并至少执行：关卡解析、碰撞核心、玩家状态转移、砖块 / 硬币一次性逻辑、敌人踩踏与伤害互斥、管道状态锁定、旗杆单次结算、基准 replay、布局差异检查。失败不能以“本地能玩”绕过。

## 缺陷等级

| 等级 | 定义 | 发布处理 |
| --- | --- | --- |
| P0 | 崩溃、无法开始 / 通关、数据损坏、合规阻断 | 立即停止候选发布 |
| P1 | 核心玩法错误、布局偏差、可重复软锁、重大无障碍失败 | 候选发布前必须修复 |
| P2 | 明显表现 / 性能问题，有替代路径 | 需评审风险和排期 |
| P3 | 轻微瑕疵或文本问题 | 可进入已知问题 |

缺陷报告包含构建号、关卡 / 调参 revision、平台浏览器、输入设备、复现步骤、期望 / 实际、录像或截图、频率和严重度。不要只写“跳跃有问题”。

## 发布候选测试

对每个 RC 执行：全自动化套件；20 次从头通关；快走、慢走、每种体型、每个入管 / 出管和旗杆高度；三种浏览器；键盘与手柄；低性能降级；无音频与无障碍矩阵；资产来源核对。结果写入发布记录，由 QA 给出明确的 Go / No-Go 建议。
