# 项目章程

状态：基线

## 目标与成功定义

交付一个可在主流桌面浏览器完整游玩的单关横版动作体验。成功不是“做出类似关卡”，而是同时满足：

- 参考审核后的空间关系、风险顺序、奖励路线和终点流程均与目标第一关一致。
- 跑跳、惯性、顶击、踩踏、受伤、进管和通关能在固定输入录像下稳定复现。
- 所有可见和可听资产原创，具有明确来源与审核记录。
- 在目标设备上清晰、稳定、可访问，不因现代特效牺牲操作判断。

## 范围

| 包含 | 不包含 |
| --- | --- |
| 标题、设置、单一正式关卡、奖励子区、结算 | 多关卡、地图、账号、联网、排行 |
| 键盘 / 手柄输入、练习模式、无障碍设置 | 多人、关卡编辑器、移动端触屏首发 |
| 原创角色 / 敌人 / 场景 / 音频 / UI | 原作名称、标志、素材、音乐和 ROM 数据 |

## 目标设备与体验预算

首发目标为 Windows / macOS / Linux 的现代 Chromium、Firefox、Safari 桌面浏览器；主视觉基准为 16:9、1920×1080、60 FPS。较低性能设备可降低粒子、后处理和远景层数，但不得降低物理 tick、输入采样或碰撞精度。移动端是后续适配项，不能反向限制首发控制方案。

## 决策原则

1. 可玩性判定优先于画面炫技；2. 可复现数据优先于设计者记忆；3. 原创表达优先于“像原作”的素材模仿；4. 自动验证优先于人工口头确认；5. 不可逆或高成本决定在预制作阶段签核。

## 角色与责任

| 角色 | 决策 / 产出 |
| --- | --- |
| 制作负责人 | 范围、优先级、里程碑放行和风险升级 |
| 玩法 / 关卡负责人 | 参考取证、布局清单、调参基线、玩法验收 |
| 工程负责人 | 架构、性能预算、数据契约、CI 与技术风险 |
| 美术 / 音频负责人 | 原创资产方向、资产清单、可读性和来源证明 |
| QA 负责人 | 测试计划、缺陷分级、回归结果和发布建议 |

同一人可兼任，但“布局记录”和“布局复核”不得由同一人单独签核。

## 变更控制

新增玩法、改变坑宽、移动实体、改写跳跃窗口、替换输入方案或修改终点流程均为受控变更。变更单应包含：目的、受影响的文档 / 数据 ID、参考依据、风险、回归用例、审批人和 `layoutRevision` / `tuningRevision`。纯原创美术替换不改变碰撞和可读性时，可走资产评审流程。
