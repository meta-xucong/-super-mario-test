# 制作、发布、合规与文档治理

状态：基线

## 里程碑与放行条件

| 里程碑 | 重点 | 放行条件 |
| --- | --- | --- |
| P0 预制作 | 参考、范围、原创方向 | 开工门槛全部关闭 |
| P1 垂直切片 | 跑跳、碰撞、首段交互 | 回放稳定，玩法负责人签核 |
| P2 全关灰盒 | 地表、奖励区、终点 | 冻结布局可完整通关 |
| P3 表现整合 | 美术、动画、音频、UI | 不改变逻辑，性能达标 |
| P4 候选发布 | QA、可访问性、合规 | P0/P1 缺陷为零，清单签核 |

每项任务必须有负责人、验收条件、关联文档 / 数据 revision、依赖与风险。里程碑演示使用可复现构建和固定 replay，不接受仅现场手动演示。

## 素材溯源与合规

建立不进运行包的素材登记表。每条记录含资产 ID、作者、创作方式、源文件位置、许可证 / 合同、第三方成分、生成式工具及输入素材声明、审核人、审核日期、替换计划。禁止提交或分发原始游戏的 ROM、截图、音频、字库、精灵图、提取数据、商标或容易造成来源混淆的名称与 Logo。对“玩法 / 关卡布局复刻”可能带来的地区差异，应在发布前获取适用地区的专业法律意见；本文件不是法律意见。

## 构建与发布

构建产物应包含版本号、提交哈希、关卡 revision、调参 revision 和资产清单哈希。发布前冻结依赖版本，记录浏览器测试矩阵、性能结果、已知问题与回滚方案。若发布后发现 P0、版权 / 商标风险或数据破坏问题，立即下架或回滚到上一批准构建。

## 文档治理

- 主规格定义“做什么”；配套文档定义“如何证明、如何实现、如何验收”。冲突时先暂停实现，由制作负责人裁决并同步更新所有受影响文档。
- 每份文档首行写明状态：`草案`、`待冻结`、`基线`、`废弃`。基线变更须在提交信息或变更日志中写明原因。
- 关卡、调参、资产和测试均用稳定 ID 与 revision 关联。没有 revision 的截图、录像和反馈不能作为验收证据。
- 所有文档以 UTF-8 Markdown 保存；引用使用相对链接；不把机密、个人信息或未授权媒体提交到仓库。

## 开发启动包

开发启动时应一次性创建：任务看板、参考登记表、冻结布局包、调参基线、资产登记表、测试用例库、缺陷模板、构建版本清单和决策日志。上述任一项缺失时，允许进行受控原型，不允许进入正式内容生产。
