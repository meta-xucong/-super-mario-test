# 开发前审计与开工门槛

状态：基线（2026-07-11）

## 审计范围

审计对象为 `super_mario_level1_spec.md` 及仓库现状，标准是“独立开发团队能否不依赖口头解释开始制作”。本次不审计任何程序实现，因为仓库尚未进入编码阶段。

## 结论

主规格的玩法边界、系统拆分、视觉升级原则和测试方向已足够作为核心设计基线；但它不能单独支持无歧义实现。最大风险不是引擎选型，而是以记忆代替可复核的原版关卡数据，导致“看起来像”而非“布局与节奏贴合”。以下配套文档已建立以关闭该风险。

| 编号 | 审计发现 | 风险级别 | 处置 | 状态 |
| --- | --- | --- | --- |
| A-01 | 没有可审计的逐格布局真源 | 阻断 | 定义取证、双人复核、布局冻结和变更流程 | 待冻结 |
| A-02 | 主规格仅给出参数类别，没有可签核的数值表 | 高 | 建立调参表、录像基准与误差容限 | 待冻结 |
| A-03 | 系统边界存在，但状态、事件和存档契约未定义 | 高 | 建立技术架构契约和事件命名规范 | 基线 |
| A-04 | 现代化美术方向未形成可交付资产规范 | 高 | 建立视觉、动画、VFX、音频的制作与验收标准 | 基线 |
| A-05 | 无障碍和设备适配未变成验收项 | 中 | 建立 UX / 无障碍矩阵 | 基线 |
| A-06 | 测试、缺陷处理和发布责任未定义 | 高 | 建立 QA、发布、素材溯源与文档治理流程 | 基线 |
| A-07 | 示例 JSON 的尺寸、坐标可能被误当最终数据 | 中 | 明确示例仅为格式，最终数据只可从冻结清单生成 | 基线 |

## 已核对的主规格优点

- 明确了“玩法 / 布局复刻”与“原创视觉 / 音频表达”的边界。
- 选择固定步长与专用平台运动控制器，能避免可变帧和通用刚体对手感的破坏。
- 覆盖角色、敌人、砖块、管道、旗杆、分数、镜头、性能和回放验证。
- 将可访问性、视觉可读性和性能列入正式要求，而非发布后补丁。

## 开工门槛

下列项目全部完成后，才可建立游戏应用、安装依赖或编写玩法代码：

- [ ] 指定合法可用的参考材料来源，并填写 [参考清单](02-reference-and-layout-freeze.md#参考清单与证据链)。
- [ ] 由记录者和复核者签署逐格布局清单；地表、奖励区、实体和触发器都有稳定 ID。
- [ ] 通过布局校验：出生点、每个入口 / 出口、终点、地面断点和可达路径无歧义。
- [ ] 以参考录像完成第一版运动测量，并冻结 `tuning-baseline` 的初始值和容限。
- [ ] 确认目标平台、最低浏览器、逻辑 tile 尺寸、设计分辨率和手柄映射。
- [ ] 视觉负责人提交原创角色、敌人、砖块、管道、旗帜和 HUD 的方向板，确认没有复制性表达。
- [ ] 建立素材登记表、任务看板、缺陷模板和 CI 所需的最低测试清单。

## 审计后的约束

1. 不得把截图、原始音频、ROM、抓取的精灵图或其可替代重绘提交到本仓库。
2. 不得在代码中硬编码个别砖块或敌人的“修正逻辑”；布局差异一律回到数据清单处理。
3. 在布局冻结前可制作技术原型，但不可宣称“关卡已还原”，也不可产出正式美术。
4. 每次修改玩法、布局、输入窗口或终点流程，必须更新主规格并重跑相应回放测试。

## 后续审计节点

M1（跑跳原型）、M3（全关可通）、M5（候选发布包）各进行一次审计。阻断问题未关闭则不得进入下一里程碑。
