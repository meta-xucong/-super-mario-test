# 《超级玛丽》第一关（1-1）现代化重制——开发设计文档

> 文档版本：1.0
> 目标平台：PC 浏览器（键鼠 / 手柄）优先，架构可扩展至桌面与移动端
> 推荐技术栈：TypeScript + Vite + WebGL 渲染器（PixiJS 或等价方案）+ Web Audio API
> 设计原则：**关卡节奏、操作手感、交互规则与原版 1-1 对齐；美术、动画、粒子、镜头、音频全部原创并以现代技术升级。**

> 开发前请先阅读 [审计与开工门槛](docs/00-readiness-audit.md)、[参考取证与布局冻结](docs/02-reference-and-layout-freeze.md) 与 [技术架构契约](docs/03-technical-architecture.md)。本文件中用于说明数据格式的坐标、宽高或实体示例不是最终关卡数据。

---

## 1. 项目定义与边界

### 1.1 一句话目标

制作一段可从标题界面开始、完整游玩到结算界面的横向卷轴动作关卡。玩家应能在数分钟内体验到与经典第一关相同的“教学—试探—掌握—冲刺—庆祝”节奏，同时获得高分辨率场景、流畅角色动画、环境特效和现代化可访问性选项。

### 1.2 必须达成

- 从左向右的单屏跟随卷轴，包含起点、问号砖教学区、首个敌人、连续跳跃区、地下奖励区入口、空中砖块谜题、中段断崖、台阶与旗杆终点。
- 跑、走、滑行、跳跃、踩敌人、顶砖、吃道具、受伤、死亡、倒计时、旗杆通关与结算均可运行。
- 关卡布局、敌人出现顺序、奖励入口位置和通关流程以原版 1-1 的参考记录为唯一游戏设计依据；不得为“好玩”任意增加平台、敌人、收集物或捷径。
- 画面与音效为原创升级资产，不使用原作的像素图、关卡图片、音乐、采样、角色名称 Logo 或直接提取的 ROM 数据。
- 60 FPS 为性能目标；在低性能设备上优先保证输入、物理和碰撞稳定，再降低特效质量。

### 1.3 不在首版范围内

- 多关卡、存档、世界地图、多人、编辑器、网络排行榜。
- 原版中的火球攻击、游泳、藤蔓关、Boss、地下完整独立关；地下区域仅作为 1-1 奖励区的受控子场景。
- 将通用刚体引擎直接用于玩家移动。平台跳跃的手感必须由专用运动控制器决定。

### 1.4 知识产权与“贴合原版”的处理

本项目采用“玩法和空间关系复刻、表达原创”的方式。保留可验证的关卡节奏和交互语法，重新设计角色外形、怪物轮廓、砖块纹理、世界观、音色、UI 文案与关卡名称。开发资料不得收录或分发 ROM、反编译代码、原始精灵图、原曲音频或可替代原始素材的逐帧重绘。发布前应完成名称、商标、素材来源和地区合规审查。

## 2. 玩家体验目标

### 2.1 三十秒体验

玩家在安全平地学会移动和跳跃；第一个巡逻敌人迫使其理解“跳过 / 踩踏 / 受伤”；第一个问号砖用清晰的视觉与声音反馈建立“向上顶击”的规则；蘑菇（项目内可使用原创名称，例如“能量孢子”）会沿地面移动，诱导玩家追逐并自然学会成长状态。

### 2.2 完整关卡节奏

| 段落 | 原版功能 | 玩家心理 | 现代化表现（不改变空间和判定） |
| --- | --- | --- | --- |
| 起点安全区 | 熟悉移动与镜头 | 放松、观察 | 远景云海、轻微风吹草动、镜头预读 |
| 首敌与首砖 | 教学第一次风险 | 紧张后获得反馈 | 敌人脚步尘粒、砖块碎屑、清晰落点影 |
| 奖励与连续砖 | 建立“探索有回报” | 好奇、兴奋 | 道具描边、可控的提示音与触觉反馈 |
| 管道奖励区 | 奖励观察力 | 惊喜、短暂休息 | 转场遮罩、地下体积光与回声 |
| 中段跳跃与断崖 | 检验速度和跳跃距离 | 专注 | 速度线极轻、镜头前瞻、失败原因可读 |
| 台阶与旗杆 | 冲刺和仪式感 | 释放、完成感 | 旗帜布料、镜头收束、烟花与原创庆祝乐句 |

### 2.3 手感优先级

1. 输入响应与落地判定；2. 跳跃弧线与惯性；3. 敌人踩踏判定；4. 镜头可读性；5. 视觉精细度。任何特效不得遮住落点、碰撞边缘或可收集物。

## 3. 玩法规格

### 3.1 输入

| 动作 | 键盘默认 | 手柄默认 | 规则 |
| --- | --- | --- | --- |
| 左 / 右移动 | A/D 或 ←/→ | 左摇杆 / 十字键 | 数字轴输入，支持缓起缓停 |
| 冲刺 | Shift | 右肩键 | 按住后提高目标速度；与跳跃组合形成远跳 |
| 跳跃 | Space / Z | 南侧按键 | 按下起跳，按住影响上升时长 |
| 暂停 | Esc | Menu | 完全冻结游戏时钟与输入边沿 |

输入层每帧只产出 `moveAxis`、`jumpPressed`、`jumpHeld`、`runHeld`、`pausePressed`。键位、手柄、触屏控制均映射到同一动作，不允许游戏逻辑直接读取键码。

### 3.2 玩家状态

```text
Small / Powered                // 体型与承伤状态
Grounded / Rising / Falling    // 运动状态
Standing / Walking / Running / Skidding
EnteringPipe / ExitingPipe / FlagSlide / CastleWalk
HurtInvincible / Dying / Dead / Cleared
```

- 小体型被敌人或危险命中后进入死亡；强化体型首次命中后缩小，并在短暂无敌窗口内闪烁。
- 敌人上方接触且玩家纵向速度向下时，判为踩踏：敌人死亡、玩家获得一次固定向上的反弹速度。
- 玩家头部撞击可顶击砖块；砖块不能被横向触发。
- 死亡、进管、触旗和结算是高优先级状态：进入后取消普通移动控制，避免状态重入。

### 3.3 运动模型

采用固定步长、确定性运动控制器。渲染可变帧，物理固定 `1/120 s`；一帧过慢时最多补算有限步数，超过部分丢弃并记录性能事件，避免“慢一帧跳得更远”。所有关卡数据使用逻辑单位，不与贴图像素绑定。

建议基准（须在试玩中以原版参考录像校准，而非盲目固定）：

| 参数 | 含义 | 调参原则 |
| --- | --- | --- |
| `walkMaxSpeed` / `runMaxSpeed` | 行走、冲刺最高水平速度 | 以固定距离助跑后的通过时间校准 |
| `groundAccel` / `airAccel` | 地面、空中加速度 | 空中仍可修正，不能像现代双跳游戏般过强 |
| `friction` / `skidDecel` | 松键阻力、反向滑行减速度 | 让反向时有可见滑行，不能立即折返 |
| `jumpVelocity` | 起跳初速度 | 以最高砖层与标准断崖为基准 |
| `jumpHoldGravityScale` | 按住跳跃时的重力倍率 | 允许短跳和长跳，设置最大按住帧数 |
| `gravity` / `fallMaxSpeed` | 下落速度 | 保证落地干脆、镜头不丢失玩家 |
| `bounceVelocity` | 踩敌反弹 | 必须能建立连续踩踏节奏 |

实现顺序：读取输入 → 水平加速/减速 → 跳跃边沿与可变跳高 → 施加重力 → X 轴扫掠碰撞 → Y 轴扫掠碰撞 → 更新接地/顶头事件 → 触发实体接触。禁止用“位置重叠后推开”的方式处理高速移动。

### 3.4 宽容机制

- **土狼时间**：离开边缘后保留很短的起跳窗口，减少视觉上已踩在边缘却无法跳的挫败。
- **跳跃缓冲**：落地前很短时间按跳，可在接地瞬间自动跳。
- **单向平台规则**：仅从上方向下落时承接；上升或横向穿过不阻挡。
- 这些机制只能改善输入容错，不能改变可达平台、跳跃高度、关卡顺序或通关时间窗。

## 4. 第一关还原规范

### 4.1 单一真源与验收方式

不要将“凭印象摆砖”当作还原。创建一份不进发布包的参考记录：合法获得的原版游玩录像、逐屏截图、人工标注的网格坐标表和操作脚本。关卡作者按该记录写入 `level-1-1.json`，评审用同一份记录逐屏比对。每次改动需更新 `layoutRevision` 与变更原因。

坐标约定：左上为 `(0, 0)`，向右为正 X，向下为正 Y；所有静态格子用整数 `tile` 坐标，实体位置使用逻辑浮点单位；碰撞尺寸以 tile 分数表示。参考表必须记录：地面断点、每个砖 / 问号砖 / 隐藏块、管道口、硬币、敌人出生点、旗杆、城堡门、奖励区入口与出口。

### 4.2 关卡分段清单

下表是实现和评审的必备清单，具体网格坐标以经审核的 `level-1-1.json` 为准。顺序、风险曲线和奖励路径不得改写。

| 顺序 | 原版空间事件 | 实现要求 | 验收点 |
| --- | --- | --- | --- |
| 01 | 左侧起点、平地、时间与生命 UI | 玩家出生朝右；镜头不倒退到关外 | 0 输入时角色稳定站立 |
| 02 | 首个巡逻地面敌人 | 固定出生 / 巡逻、边缘规则符合原版参考 | 可跳过、可踩、横向接触可受伤 |
| 03 | 第一组低位互动砖与问号砖 | 包含普通砖、奖励砖、顶击动画 | 从下方顶击一次后状态正确 |
| 04 | 首个成长道具 | 从砖上弹出，沿地面移动、可反弹 | 道具不穿墙、不在生成瞬间被吞掉 |
| 05 | 连续砖块 / 硬币教学区 | 保持原版高度和间距 | 玩家可用短跳、长跳分别通过 |
| 06 | 第一根可进入管道及地下奖励区 | 仅在正确位置按下进入 | 进入、收集、返回均不丢失状态 |
| 07 | 中段开阔跑道与敌人组合 | 敌人时序、砖列和管道排列与参考一致 | 全速跑与保守跳均有可读路线 |
| 08 | 隐藏块与空中硬币谜题 | 隐藏块不可用发光轮廓泄题 | 被顶击后出现正确奖励 / 反馈 |
| 09 | 首次明确断崖与高低台 | 坑宽、台阶高差不可现代化缩短 | 标准助跑跳可稳定通过 |
| 10 | 末段连续台阶与敌人 | 台阶形成上升节奏，敌人不能被特效遮挡 | 满速与减速两种策略均符合参考 |
| 11 | 旗杆、落地、城堡门 | 触旗后锁定玩法并执行仪式动画 | 旗杆高度决定奖励，之后走入城门 |

### 4.3 地下奖励区

地下区是奖励分支，不是新增自由探索地图：入口、出口、砖列与硬币序列严格来自参考记录。该区需要独立背景层和音频混响，但共享同一物理与时间系统。进入和离开期间：

- 保存 `returnAnchorId`、玩家体型、剩余时间、分数、已触发方块和收集状态。
- 使用黑场或原创转场遮罩；禁止在加载过程中继续消耗时间或接受移动输入。
- 出口回到指定地表锚点，镜头、玩家朝向和高度与参考一致。

### 4.4 终点流程

触及旗杆碰撞体后，根据触点相对于旗杆有效高度换算奖励等级；禁止再次输入；角色滑杆、落地、旗帜降下、走向城堡门、结算时间奖励、显示完成画面。所有计分变化写入事件队列，由 UI 订阅，不能在动画中直接改分，避免跳帧造成重复加分。

## 5. 关卡数据与工具链

### 5.1 文件结构

```text
src/
  app/                 # 启动、场景切换、依赖装配
  core/                # 固定时钟、事件、随机种子、存档接口
  input/               # 键鼠、手柄、动作映射
  gameplay/
    player/            # 控制器、状态机、动画意图
    entities/          # 敌人、方块、道具、旗杆、管道
    physics/           # tile sweep、AABB、触发器、单向平台
    rules/             # 分数、时间、生命、通关
  levels/
    level-1-1.json     # 运行时关卡真源
    level-1-1.meta.json# 参考版本、校验和、测试锚点
  render/              # 图层、相机、材质、粒子、后处理
  audio/               # 混音、事件到音效映射、无障碍字幕
  ui/                  # HUD、暂停、设置、结算
assets/
  art/ audio/ fonts/ shaders/
tests/
  unit/ integration/ replay/ visual/
docs/
  reference/           # 仅可合法分发的标注与说明，不放原始受保护资产
```

### 5.2 TileMap 数据约定

使用 JSON 或 Tiled 导出的 JSON，拆分视觉和逻辑层。视觉层可替换为高精资源，逻辑层永远是简洁、可审计的网格数据。

```json
{
  "id": "level-1-1",
  "layoutRevision": 1,
  "tileSize": 16,
  "bounds": { "width": 212, "height": 15 },
  "layers": ["background", "decor", "solid", "oneWay", "hazard", "logic"],
  "spawn": { "x": 2.5, "y": 11, "facing": 1 },
  "entities": [
    { "id": "goomba-001", "type": "walker", "x": 22, "y": 12, "patrol": "reference" },
    { "id": "pipe-001", "type": "pipe", "x": 58, "y": 11, "destination": "bonus-a" }
  ],
  "completion": { "type": "flagpole", "entityId": "flagpole-001" }
}
```

示例中的尺寸和坐标只说明格式，提交前必须由参考标注表生成真实数据，不能把示例值当作最终布局。`id` 一经发布不得变更，以便录像回放、测试和存档定位。

### 5.3 实体定义

| 类型 | 必需字段 | 行为 |
| --- | --- | --- |
| `walker` | 出生点、朝向、速度、碰撞轮廓 | 地面巡逻、碰墙转向、可踩踏 |
| `questionBlock` | 内容物、已使用状态 | 仅可下顶；弹跳、出物、变空 |
| `breakableBlock` | 强度 / 可破坏条件 | 小体型顶击不破；强化体型可破碎 |
| `powerup` | 种类、初始方向 | 出现后离开方块，受地形约束移动 |
| `coin` | 分值、收集状态 | 触碰即收集，播放不遮挡游戏的反馈 |
| `pipe` | 入口方向、目标锚点 | 条件满足后接管角色并执行转场 |
| `flagpole` | 高度、奖励区间 | 触发终点序列 |

### 5.4 校验工具

- `validate-level`：检查越界 tile、重叠实体、不可达出口、重复 ID、未引用目的地。
- `render-level`：输出无美术的碰撞图和坐标网格，供逐屏审查。
- `diff-layout`：与批准的基线 JSON 比对，输出新增、删除、移动的对象。
- `play-replay`：播放确定性输入录像，对比关键帧位置、状态、分数和剩余时间。

## 6. 物理与碰撞实现

### 6.1 碰撞层级

1. Tile 静态碰撞（实心、单向、危险、斜坡如有）；2. 动态实体 AABB；3. 触发器；4. 视觉效果。视觉网格没有碰撞权，逻辑碰撞层是唯一判定来源。

每个可移动实体维护 `previousPosition`、`position`、`velocity`、`collider`、`grounded`、`contactNormal`。X/Y 分离扫掠时记录首次命中法线，避免角落处重复触发顶砖或踩敌。

### 6.2 判定优先级

同一物理步内：终点触发 > 死亡危险 > 进管条件 > 敌人踩踏 / 受伤 > 收集物 > 方块顶击。终点一旦触发，丢弃其余可变更游戏状态的接触事件。每类事件用 `(tick, sourceId, targetId)` 去重。

### 6.3 死亡与重生

掉入死亡边界、敌人命中或其他危险触发 `Dying`。立即禁止控制、停止相机前瞻；死亡动画结束后扣除生命并重新载入当前关卡初始快照。首版不提供中间检查点，符合第一关的完整挑战结构。重开必须重置可破砖、方块、敌人和硬币；HUD 的生命和累计分数按规则保留。

## 7. 渲染、美术与特效升级

### 7.1 视觉方向

采用“绘本式微缩世界 + 清晰动作剪影”，而不是把低分辨率像素简单放大。使用原创角色：圆润、易辨认的冒险者；敌人使用不同于原作的面部比例、轮廓和配色。世界色彩以晴天蓝、温暖土色和高饱和植被为基础，确保可玩对象与背景的明度对比达到可读标准。

### 7.2 分层

```text
天空渐变 / 云雾（最远）
远景山体与缓慢视差
中景植被、建筑剪影
可碰撞地形与互动砖
玩家、敌人、道具
前景草叶、尘粒、局部遮挡（不得遮挡落点）
HUD / 暂停 UI
```

- 使用 3–5 层视差；远景移动比例低，近景比例高。相机未移动时背景也只做极弱环境动画，避免视觉噪声。
- 角色采用骨骼或高帧序列动画：待机、起跑、跑步、急停、起跳、下落、踩踏、受伤、进管、触旗分别有独立姿态。
- 地形采用模块化边缘、法线细节和轻微动态光照，但碰撞边界永远按逻辑 tile，不随贴图凸起改变。

### 7.3 特效预算与规则

| 事件 | 特效 | 性能与可读性约束 |
| --- | --- | --- |
| 跑步 / 落地 | 方向性尘粒、压草 | 不超过角色脚部附近小范围 |
| 顶砖 | 砖块上抬、碎屑、短暂冲击环 | 不遮挡块上方的道具 |
| 收集 | 小型闪点、数字飘字 | UI 合并同帧事件，避免满屏数字 |
| 踩敌 | 轻微形变、星屑 | 击杀轮廓和玩家反弹必须清晰 |
| 通关 | 旗帜布料、少量彩带、远景烟花 | 完成状态后播放，可在设置中降级 |

提供“减少动态效果”“关闭屏幕震动”“高对比度轮廓”“色弱配色”选项。镜头震动默认极轻且不影响物理相机坐标。

### 7.4 相机

相机以玩家为中心偏右前瞻，向右移动时平滑扩大前方可见范围；向左回退时减少前瞻但不猛拉。使用阻尼和死区，绝不让镜头抖动传递给碰撞坐标。关卡左、右边界夹紧；管道和终点进入脚本相机。设计分辨率支持 16:9 主目标，同时在超宽屏扩展背景而不是提前展示关卡机关。

---

## 8. 音频设计

- 原创音乐以轻快的木管、马林巴、拨弦为主；结构上随跑动和通关增强，但旋律、和声与节奏型不得复制原曲。
- 音效按“材质 + 动作”组织：土路脚步、砖石顶击、空心问号砖、软体敌人、金属旗杆、管道回响。
- 以游戏事件驱动音频：同一帧的大量硬币声做限频与聚合；暂停、死亡、通关切换不同总线状态。
- 提供主音量、音乐、效果、动态范围、音效字幕开关；音效字幕如“[砖块顶击]”“[敌人靠近]”显示在不遮挡角色的位置。

---

## 9. UI、分数与可访问性

HUD 固定显示得分、硬币数、世界 / 关卡标识、剩余时间与生命数。文字与图形使用原创设计，不模仿原作 UI 字形和排版。分数系统统一从 `GameEvent` 消费，事件示例：`CoinCollected`、`EnemyStomped`、`BlockHit`、`GoalReached`、`TimeConverted`。

可访问性最低要求：可重绑键位、完整手柄导航、暂停后不丢输入、可关闭屏闪 / 镜头震动、字幕、色弱模式、高对比度交互物轮廓、文字不只依赖颜色传达状态。对可重玩性，提供“练习模式”作为独立选项：允许从指定分段开始但不计入正式成绩；默认正式模式保持原关流程。

## 10. 工程架构与质量标准

### 10.1 模块边界

- `GameSimulation`：仅保存可序列化的游戏状态，禁止直接调用渲染 / DOM / 音频。
- `Renderer`：读取模拟状态插值后渲染；资源失败时显示占位但不改变物理。
- `AudioDirector`：消费事件，做去重、限频与混音，不反向写游戏状态。
- `SceneDirector`：管理标题、游戏、暂停、结算场景和资源生命周期。
- `LevelRepository`：解析、校验并加载关卡；运行时禁止静默修正非法数据。

### 10.2 性能指标

| 指标 | 目标 |
| --- | --- |
| 游戏模拟 | 120 Hz 固定步；常规帧不丢步 |
| 主目标渲染 | 1920×1080、60 FPS |
| 首次可玩加载 | 关键资源优先，非关键装饰可延迟 |
| 内存 | 切关或重开后无持续增长 |
| Draw Call | 图集 / 批处理后稳定在预算内，粒子可动态降级 |

使用对象池管理硬币闪点、尘粒、碎屑和飘字；纹理图集按场景拆分；所有资源拥有明确销毁路径。开发模式显示 FPS、固定步数、碰撞盒、实体数量、相机边界和事件日志。

### 10.3 错误处理

资源加载失败、关卡数据无效、手柄断连均必须显示可理解的恢复提示，不能白屏或吞异常。生产环境上报匿名错误栈、浏览器信息、游戏版本和关卡 `layoutRevision`，不采集不必要的个人信息。

## 11. 测试与验收

### 11.1 自动化测试

- 单元测试：加速度、短跳 / 长跳、土狼时间、扫掠碰撞、踩踏方向、方块一次性状态、计分去重。
- 关卡数据测试：所有入口有出口、所有实体 ID 唯一、终点存在、玩家出生点不嵌入碰撞体、关键路线可达。
- 回放测试：录制输入种子和每 30 tick 的快照哈希；CI 中在无渲染模式重放，比对位置、状态、分数、时间与事件序列。
- 视觉回归：固定关键帧截屏，比较布局、相机边界、HUD 遮挡与粒子上限；允许美术迭代的区域需建立掩码。

### 11.2 人工试玩矩阵

| 场景 | 验收结果 |
| --- | --- |
| 慢走完成 | 可不冲刺、不依赖隐藏技巧而完成 |
| 全速冲刺 | 不穿墙、不漏碰撞、不因镜头滞后死亡 |
| 每种体型 | 砖块破坏、受伤与狭窄区域状态正确 |
| 道具 / 敌人边缘接触 | 踩踏与受伤结果稳定，无双重计分 |
| 进出奖励区 | 时间、状态、镜头、音乐和锚点连续 |
| 旗杆各高度 | 奖励正确，只结算一次 |
| 低性能 / 焦点丢失 | 暂停、恢复后不快进、不积累输入 |
| 键盘、手柄、触屏 | 同一动作手感和暂停逻辑一致 |

### 11.3 发布门槛

1. 完整正式流程连续通关 20 次，无阻断 Bug；2. 关卡逐屏审查无未经批准的布局偏差；3. 自动化测试与回放测试全绿；4. 60 FPS 主目标设备稳定；5. 素材来源、授权与商标检查通过；6. 无障碍设置可用且回归通过。

| 阶段 | 交付物 | 完成定义 |
